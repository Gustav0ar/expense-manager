## Why this matters

`drizzle/meta/_journal.json` records migration `0009` with a timestamp older than `0008`. Drizzle 0.45.2 decides whether to run a migration by comparing the latest database migration timestamp with each journal timestamp. A database already migrated through `0008` therefore skips `0009`, while a fresh database applies it. The skipped migration adds `user_mfa_config.last_used_totp_counter`, which current MFA code reads and writes.

The repair must be forward-only. Editing the old timestamp would cause databases that already applied `0009` to try it again and fail.

## Steps

### 1. Create a forward-only idempotent repair

Use Drizzle Kit's custom migration flow so the new journal entry has a timestamp greater than `0014` and a snapshot of the current schema. The SQL must add `last_used_totp_counter bigint` with `IF NOT EXISTS`. Inspect the generated snapshot and ensure it contains all schema objects introduced through `0014`.

Do not change the timestamp or SQL of `0009`.

**Verify**: parse `_journal.json` in a small Node command and assert every new entry after `0014` has a strictly greater `when`; confirm the new SQL contains `ADD COLUMN IF NOT EXISTS`.

### 2. Add a real upgrade-path test

Create `scripts/test-migration-upgrade.sh`. It must:

1. create a uniquely named temporary PostgreSQL database;
2. guarantee cleanup with a shell trap;
3. apply only migrations `0000`–`0008` using a temporary migration directory and truncated journal;
4. apply the complete current migration set to the same database;
5. query `information_schema.columns` and assert `user_mfa_config.last_used_totp_counter` exists;
6. run the complete migration set a second time and assert it remains successful.

Never point the script at the normal development database. Refuse to run if the temporary database name does not match the script's generated prefix.

Add `test:migrations` to `package.json`, invoke it in CI after the normal fresh migration, and document it in `docs/development.md`.

**Verify**: run `pnpm test:migrations` twice; both invocations exit 0 and leave no temporary database.

### 3. Validate fresh and upgraded databases

Run migrations twice against the normal disposable dev database, then run type-check, unit tests, E2E and build through `pnpm verify`.

**Verify**: `pnpm verify` exits 0.
