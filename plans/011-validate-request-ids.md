## Why this matters

The server accepts any `X-Request-Id` header and copies it into response headers, structured logs and trace attributes. Unbounded attacker-controlled identifiers increase log/trace cardinality and blur which IDs are generated by trusted infrastructure.

## Current state

- `src/hooks.server.ts:47-49` and `118-120`: directly trust the request header.
- `src/hooks.server.ts:156-171`: request ID enters structured error logs.
- `src/lib/server/observability/tracing.ts:129-135`: request ID becomes a span attribute.

## Steps

### 1. Define a bounded contract

Accept only a conservative ASCII identifier format and maximum length (for example UUID/ULID-compatible characters up to 64). Generate an internal UUID when absent or invalid. If preserving a valid external correlation ID separately, name it explicitly and never use it as the authoritative internal ID.

**Verify**: unit tests cover valid, missing, oversized, control-character and Unicode inputs.

### 2. Apply consistently

Ensure one internal ID is created once, used in response/log/trace, and cannot be replaced later in the hook chain. Document proxy correlation behavior.

**Verify**: infrastructure/E2E test sends valid and invalid headers and asserts bounded response IDs without response-splitting or raw invalid content.

### 3. Run all gates

**Verify**: `pnpm verify` passes.
