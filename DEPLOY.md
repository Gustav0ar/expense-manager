# Expense Manager Deployment

This is the single source of truth for publishing this public repository to a
private VPS behind Traefik without committing personal data, hostnames, IP
addresses, passwords or SSH keys.

The repository is safe to keep public as long as production values stay in:

- GitHub environment secrets and variables.
- Your private DNS, Traefik and server configuration.
- The private `.env` file generated on the VPS by the deploy workflow.

Do not commit `.env`, real domains, private IPs, usernames, SMTP credentials,
database passwords, SSH keys, backup locations or customer/company data.

## GitHub Configuration

Create a protected GitHub environment named `production`.

Recommended environment protection:

- Required reviewers: enabled.
- Deployment branches: restrict to your default production branch.
- Secrets: environment-scoped, not repository-wide.

### Environment Secrets

Set these in `Settings -> Environments -> production -> Secrets`.

| Secret              | Example                                  | Notes                                                                  |
| ------------------- | ---------------------------------------- | ---------------------------------------------------------------------- |
| `VPS_HOST`          | `203.0.113.10`                           | VPS host or DNS name. Keep private.                                    |
| `VPS_USERNAME`      | `expense-manager-deploy`                 | Dedicated deploy user.                                                 |
| `VPS_PORT`          | `22`                                     | Optional; defaults to `22`.                                            |
| `VPS_SSH_KEY`       | `-----BEGIN OPENSSH PRIVATE KEY-----...` | Private key generated by the setup script.                             |
| `TAILSCALE_AUTHKEY` | `tskey-auth-...`                         | Optional. Required when `VPS_HOST` is a Tailscale IP or MagicDNS name. |
| `VPS_ENV_FILE`      | Full production `.env` content           | Optional multiline secret copied to the VPS at deploy time only.       |

If the VPS SSH port is private and reachable only through Tailscale, set
`VPS_HOST` to the VPS Tailscale IP or MagicDNS name and set
`TAILSCALE_AUTHKEY`. Use an ephemeral, reusable, pre-approved auth key tied to a
restricted tag such as `tag:ci`.

If `VPS_ENV_FILE` is not set, the deploy workflow builds the VPS `.env` from
individual protected environment secrets:

| Secret                  | Notes                                                                         |
| ----------------------- | ----------------------------------------------------------------------------- |
| `DOMAIN_NAME`           | Bare production hostname.                                                     |
| `ORIGIN`                | Full production origin.                                                       |
| `BETTER_AUTH_SECRET`    | High-entropy app secret.                                                      |
| `POSTGRES_PASSWORD`     | Database password.                                                            |
| `RESTIC_REPOSITORY`     | Remote off-VPS restic repository. Required when `BACKUP_ENABLED=true`.        |
| `RESTIC_PASSWORD`       | High-entropy restic repository password. Required when `BACKUP_ENABLED=true`. |
| `SENDER_API_TOKEN`      | Sender transactional email API token.                                         |
| `SENDER_FROM`           | Verified sender address, optionally with name.                                |
| `TRUSTED_ORIGINS`       | Optional comma-separated extra origins.                                       |
| `AWS_ACCESS_KEY_ID`     | Required only for S3-compatible restic.                                       |
| `AWS_SECRET_ACCESS_KEY` | Required only for S3-compatible restic.                                       |
| `AWS_DEFAULT_REGION`    | Optional S3-compatible region.                                                |
| `SMTP_*`                | Optional SMTP fallback values.                                                |

### Environment Variables

Set these in `Settings -> Environments -> production -> Variables`.

| Variable                | Default                    | Notes                                                                                  |
| ----------------------- | -------------------------- | -------------------------------------------------------------------------------------- |
| `DEPLOY_PATH`           | `/opt/expense-manager`     | Directory on the VPS that stores compose files and `.env`.                             |
| `COMPOSE_COMMAND`       | `docker compose`           | Use `podman compose` only if your VPS supports it.                                     |
| `REGISTRY`              | `ghcr.io`                  | Container registry.                                                                    |
| `IMAGE_OWNER_LOWERCASE` | `<github-owner-lowercase>` | Lowercase GitHub user or organization that owns the package.                           |
| `CONTAINER_PLATFORMS`   | `linux/amd64,linux/arm64`  | Optional image platforms. Set to the VPS architecture only for faster private deploys. |
| `BACKUP_ENABLED`        | `true`                     | Set to `false` only to bootstrap without remote backups.                               |

Avoid storing personal values in variables if you do not want them visible to
collaborators. Use secrets for hostnames, IPs, usernames and every application
environment value. `VPS_ENV_FILE` is the recommended place for all production
app configuration.

## First-Time VPS Setup

Run as root on the VPS.

```bash
# Install Docker if needed.
curl -fsSL https://get.docker.com | sh

# Download the setup helper from your public repository.
curl -fsSL \
  https://raw.githubusercontent.com/<owner>/<repo>/<branch>/scripts/setup-vps-deploy-user.sh \
  -o /tmp/setup-vps-deploy-user.sh

# Create the deploy user and directory.
sudo bash /tmp/setup-vps-deploy-user.sh /opt/expense-manager expense-manager-deploy

# Copy this full private key into the GitHub production VPS_SSH_KEY secret.
sudo cat /root/.ssh-key-handoff/expense-manager-deploy.key

# Ensure the external Traefik network exists.
docker network create web 2>/dev/null || true
```

The deploy user is added to the `docker` group and owns `DEPLOY_PATH`. Log out
and back in if you test that account interactively before the group membership is
active.

## Production Environment

Use individual protected environment secrets for normal deployments. If you
prefer a single multiline secret, create `VPS_ENV_FILE` with the full production
`.env` content instead. When `VPS_ENV_FILE` is set, it takes precedence over the
individual app secrets.

The deploy workflow validates required keys, copies the generated file to the
VPS through SSH, installs it as `DEPLOY_PATH/.env` with mode `600` and then runs
deployment.

The workflow does not print the secret. GitHub masks environment secrets in job
logs, and the generated file is not stored as an artifact.

Use this template only if you choose the single `VPS_ENV_FILE` mode:

```env
PUBLIC_APP_NAME=Expense Manager
DOMAIN_NAME=finance.example.com
ORIGIN=https://finance.example.com
TRUSTED_ORIGINS=

BETTER_AUTH_SECRET=<openssl rand -base64 32>
REQUIRE_EMAIL_VERIFICATION=true
ALLOW_REGISTRATION=false

POSTGRES_DB=expense_manager
POSTGRES_USER=expense_manager
POSTGRES_PASSWORD=<long-random-database-password>
DB_POOL_MAX=5
TRUST_PROXY_HEADERS=true

BACKUP_ENABLED=true
RESTIC_REPOSITORY=s3:s3.example.com/expense-manager-backups
RESTIC_PASSWORD=<long-random-restic-password>
RESTIC_HOST=expense-manager
RESTIC_INIT_REPOSITORY=true
RESTIC_KEEP_DAILY=7
RESTIC_KEEP_WEEKLY=4
RESTIC_KEEP_MONTHLY=12
RESTIC_CHECK_AFTER_BACKUP=false
AWS_ACCESS_KEY_ID=<remote-backup-access-key>
AWS_SECRET_ACCESS_KEY=<remote-backup-secret-key>
AWS_DEFAULT_REGION=

REGISTRY=ghcr.io
IMAGE_OWNER_LOWERCASE=<github-owner-lowercase>

TRAEFIK_NETWORK=web
TRAEFIK_HTTP_ENTRYPOINT=web
TRAEFIK_ENTRYPOINT=websecure
TRAEFIK_CERT_RESOLVER=le

APP_MEM_LIMIT=512m
APP_CPUS=1.0
POSTGRES_MEM_LIMIT=1g
POSTGRES_CPUS=1.0
BACKUP_MEM_LIMIT=256m
BACKUP_CPUS=0.5

SMTP_PASSWORD=

```

`DOMAIN_NAME` must be the bare hostname only. Do not include `https://`, a path
or a port. `ORIGIN` is the full browser origin.

Use Sender in production if users need password reset, invitations or email
verification. See [`docs/email.md`](docs/email.md) for the Sender setup.

The workflow preserves deploy-generated state keys from the existing VPS `.env`
when it refreshes the file from GitHub secrets:

- `IMAGE_TAG`
- `PREVIOUS_IMAGE_TAG`
- `LAST_SUCCESSFUL_IMAGE_TAG`
- `LAST_SUCCESSFUL_DEPLOY_SHA`
- `LAST_PREDEPLOY_BACKUP`
- `LAST_ROLLBACK_IMAGE_TAG`
- `LAST_ROLLBACK_AT`
- `LAST_ROLLBACK_REASON`

These keys are operational state used by rollback. You do not need to manage
them in GitHub secrets.

## Deploy Flow

1. Push to `main` or `master`, or trigger the `Containers` workflow manually.
2. The `Containers` workflow builds and publishes:
   - `ghcr.io/<owner>/expense-manager-app:<tag>`
   - `ghcr.io/<owner>/expense-manager-migrate:<tag>`
   - `ghcr.io/<owner>/expense-manager-backup:<tag>`
3. The `Deploy` workflow waits for `CI`, `Security` and `Containers`.
4. The `production` environment requires approval before SSH is used.
5. On the VPS, `scripts/deploy-vps.sh`:
   - Downloads `docker-compose.traefik.yml`, `scripts/backup.sh` and
     `docker/postgres/init.sql` from the exact commit being deployed.
   - Persists current, previous and successful image metadata in the private
     `.env`.
   - Starts Postgres and waits for health.
   - Creates and validates a pre-deploy Postgres backup when an existing
     database is running.
   - Runs Drizzle migrations in the migration image.
   - Starts the app and remote backup services.
   - Verifies `/api/health` and `/` through Traefik.
   - Rolls app containers back to the previous image tag automatically when
     migration, restart, application healthcheck or public smoke checks fail.
   - Prunes dangling images.

The public `deploy-summary` artifact contains only service name, image tag,
commit SHA, image owner and a non-reversible host fingerprint.

## Rollback

Rollback has two layers:

- Image rollback: safe to automate. It moves the app and backup containers back
  to a previous image tag and re-runs health and public smoke checks.
- Database restore: intentionally manual. Restoring a pre-deploy dump can remove
  records written after deployment started, so it requires an explicit
  confirmation string.

During deploy, `scripts/deploy-vps.sh` records the previous image tag in
`PREVIOUS_IMAGE_TAG`, creates `LAST_PREDEPLOY_BACKUP` when Postgres already
exists and records `LAST_SUCCESSFUL_IMAGE_TAG` after a successful smoke test. If
the deploy fails after the previous image is known, the script automatically
rolls containers back to `PREVIOUS_IMAGE_TAG`. It does not restore the database
automatically.

For a manual rollback through GitHub:

1. Open `Actions -> Rollback -> Run workflow`.
2. Leave `image_tag` empty to use `PREVIOUS_IMAGE_TAG` from the private VPS
   `.env`, or enter a specific tag such as `sha-abcdef1`.
3. Leave `restore_database_backup` empty for an image-only rollback.
4. To also restore a pre-deploy dump, enter a path such as
   `backups/predeploy/expense_manager-YYYYMMDDTHHMMSSZ-abcdef1.dump` and set
   `confirm_database_restore` to exactly `restore-database`.
5. Approve the protected `production` environment.

Manual rollback from the VPS uses the same script:

```bash
cd /opt/expense-manager

# Image-only rollback to PREVIOUS_IMAGE_TAG from .env.
DEPLOY_PATH=/opt/expense-manager \
COMPOSE_CMD="docker compose" \
bash scripts/rollback-vps.sh

# Image rollback plus explicit database restore from a pre-deploy dump.
IMAGE_TAG=sha-abcdef1 \
RESTORE_DATABASE_BACKUP=backups/predeploy/expense_manager-YYYYMMDDTHHMMSSZ-abcdef1.dump \
CONFIRM_DATABASE_RESTORE=restore-database \
bash scripts/rollback-vps.sh
```

If a migration was destructive or incompatible with the previous app version,
use the database restore option. If the issue is only application startup,
configuration or runtime behavior, prefer image-only rollback first.

## Traefik Expectations

The compose labels assume:

- External Docker network: `web`
- HTTP entrypoint: `web`
- HTTPS entrypoint: `websecure`
- Certificate resolver: `le`

Change `TRAEFIK_NETWORK`, `TRAEFIK_HTTP_ENTRYPOINT`, `TRAEFIK_ENTRYPOINT` and
`TRAEFIK_CERT_RESOLVER` in the VPS `.env` if your Traefik instance uses
different names.

Traefik terminates TLS and forwards traffic to the `app` container on port
`3000`. The application trusts proxy headers only because it is attached to the
private Traefik network and is not published directly to the internet.

## Backup And Restore

Production backups are remote by default. The `backup` service runs daily,
creates temporary files inside the container, uploads them to an encrypted
`restic` repository and then removes local temporary files.

If you do not have a remote backup provider yet, set `BACKUP_ENABLED=false` as a
GitHub production environment variable. This disables the daily remote backup
service and lets the deploy continue without `RESTIC_REPOSITORY` or
`RESTIC_PASSWORD`. Use this only for bootstrap or non-critical data. It is not a
disaster-recovery plan.

The remote repository receives:

- Postgres custom-format dumps.
- Upload archive tarballs.
- SHA-256 checksums.

Use an S3-compatible bucket, Backblaze B2, SFTP, REST server or another restic
backend that is not stored on the same VPS. For S3-compatible storage, set
`RESTIC_REPOSITORY`, `RESTIC_PASSWORD`, `AWS_ACCESS_KEY_ID`,
`AWS_SECRET_ACCESS_KEY` and optionally `AWS_DEFAULT_REGION` in the private VPS
`.env`.

The deploy script also creates a pre-deploy dump under
`backups/predeploy/` before migrations when an existing Postgres container is
running. That local file is only a short-term rollback aid for a deploy; it is
not the disaster-recovery backup.

List remote snapshots:

```bash
docker compose run --rm --no-deps --entrypoint restic backup snapshots
```

Restore a remote snapshot to a temporary restore directory:

```bash
docker compose run --rm --no-deps \
  --entrypoint restic \
  -v "$(pwd)/restore:/restore" \
  backup restore latest --target /restore
```

Restore Postgres first:

```bash
docker compose stop app
docker compose exec -T postgres pg_restore \
  -U "$POSTGRES_USER" \
  -d "$POSTGRES_DB" \
  --clean \
  --if-exists \
  /restore/path/from/restic/expense_manager_YYYYMMDDTHHMMSSZ.dump
```

Restore uploads matching the same timestamp:

```bash
docker compose run --rm --no-deps \
  -v "$(pwd)/restore:/restore:ro" \
  app sh -lc 'rm -rf /app/uploads/* && tar -C /app/uploads -xzf /restore/path/from/restic/uploads_YYYYMMDDTHHMMSSZ.tar.gz'
docker compose up -d app
```

Always validate restores on a separate database before relying on them in
production.

## Public Repository Safety Checklist

- `.env` is ignored and never committed.
- Production values are generic in docs and examples.
- SSH keys are generated on the VPS and stored only in GitHub secrets.
- The deploy workflow does not print the real host value.
- The app is not directly published with Docker ports in the Traefik compose.
- Public route checks use the private VPS `.env` domain at deploy time.
- E2E tests use synthetic users and example domains only.
