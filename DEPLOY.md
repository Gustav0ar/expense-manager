# Expense Manager Deployment

This is the single source of truth for publishing this public repository to a
private VPS behind Traefik without committing personal data, hostnames, IP
addresses, passwords or SSH keys.

The repository is safe to keep public as long as production values stay in:

- GitHub environment secrets and variables.
- Your private DNS, Traefik and server configuration.
- The private `.env` file generated on the VPS by the deploy workflow.

Do not commit `.env`, real domains, private IPs, usernames, SMTP credentials,
database passwords, SSH keys, backup locations or customer/company data.

## GitHub Configuration

Create a protected GitHub environment named `production`.

Recommended environment protection:

- Required reviewers: enabled.
- Deployment branches: restrict to your default production branch.
- Secrets: environment-scoped, not repository-wide.

### Environment Secrets

Set these in `Settings -> Environments -> production -> Secrets`.

| -------------------------- | ----------------------------------- | ------------------------------------------------------------------------------------------------------------------------- |

| `VPS_PORT`                 | `22`                                | Optional; defaults to `22`.                                                                                               |

do not trust a fingerprint obtained through the same network connection it is
meant to authenticate:

```bash
ssh-keygen -lf /etc/ssh/ssh_host_ed25519_key.pub -E sha256
```

Avoid storing personal values in variables if you do not want them visible to
collaborators. Use secrets for hostnames, IPs, usernames and every application
environment value. `VPS_ENV_FILE` is the recommended place for all production
app configuration.

## First-Time VPS Setup

Run as root on the VPS.

```bash
# Install Docker if needed.
curl -fsSL https://get.docker.com | sh

# Download the setup helper from your public repository.
curl -fsSL \
  https://raw.githubusercontent.com/<owner>/<repo>/<branch>/scripts/setup-vps-deploy-user.sh \
  -o /tmp/setup-vps-deploy-user.sh

# Create the deploy user and directory.
sudo bash /tmp/setup-vps-deploy-user.sh /opt/expense-manager expense-manager-deploy

# Copy this full private key into the GitHub production VPS_SSH_KEY secret.
sudo cat /root/.ssh-key-handoff/expense-manager-deploy.key

# Ensure the external Traefik network exists.
docker network create web 2>/dev/null || true
```

The deploy user is added to the `docker` group and owns `DEPLOY_PATH`. Log out
and back in if you test that account interactively before the group membership is
active.

## Production Environment

Use individual protected environment secrets for normal deployments. If you
prefer a single multiline secret, create `VPS_ENV_FILE` with the full production
`.env` content instead. When `VPS_ENV_FILE` is set, it takes precedence over the
individual app secrets.

The deploy workflow validates required keys, copies the generated file to the
VPS through SSH, installs it as `DEPLOY_PATH/.env` with mode `600` and then runs
deployment.

The workflow does not print the secret. GitHub masks environment secrets in job
logs, and the generated file is not stored as an artifact.

### Rotate the application secret without stranding invitations

Queued invitation tokens are encrypted with a key derived from the application
secret. Rotate with an overlap:

1. In the protected `production` environment, set
   `BETTER_AUTH_SECRET_PREVIOUS` to the exact current value of
   `BETTER_AUTH_SECRET`.
2. Replace `BETTER_AUTH_SECRET` with a newly generated high-entropy value, then
   run the normal reviewed deployment. The workflow never writes either value to
   command output. On the VPS, `deploy-vps.sh` writes separate private Compose
   secret files and mounts the previous one at
   `/run/secrets/better_auth_secret_previous`.
3. Verify `/api/health` and accept a queued pre-rotation invitation.
4. Keep the old value for at least seven days, or explicitly resend all remaining
   pending invitations.
5. Delete the protected `BETTER_AUTH_SECRET_PREVIOUS` secret and deploy again.
   The optional secret file becomes empty and the application drops the old key.

If `VPS_ENV_FILE` mode is used, perform the same steps by keeping both keys in
the private file during overlap and removing only
`BETTER_AUTH_SECRET_PREVIOUS` afterward. Never remove the old value before the
new container has successfully started with both keys.

The workflow preserves deploy-generated state keys from the existing VPS `.env`
when it refreshes the file from GitHub secrets:

- `IMAGE_TAG`
- `PREVIOUS_IMAGE_TAG`
- `LAST_SUCCESSFUL_IMAGE_TAG`
- `LAST_SUCCESSFUL_DEPLOY_SHA`
- `LAST_PREDEPLOY_BACKUP`
- `LAST_ROLLBACK_IMAGE_TAG`
- `LAST_ROLLBACK_AT`
- `LAST_ROLLBACK_REASON`

These keys are operational state used by rollback. You do not need to manage
them in GitHub secrets.

## Deploy Flow

1. Push to `main` or `master`, or trigger the `Containers` workflow manually.
2. The `Containers` workflow builds and publishes:
   - `ghcr.io/<owner>/expense-manager-app:<tag>`
   - `ghcr.io/<owner>/expense-manager-migrate:<tag>`
   - `ghcr.io/<owner>/expense-manager-backup:<tag>`
     Each image is built with SBOM and provenance metadata, and GitHub build
     provenance attestations are pushed to the registry for the published image
     digest.
3. The `Deploy` workflow waits for `CI`, `Security` and `Containers`.
4. The `production` environment requires approval before SSH is used.
5. On the VPS, `scripts/deploy-vps.sh`:
   - Downloads `docker-compose.traefik.yml`, `scripts/backup.sh` and
     `docker/postgres/init.sql` from the exact commit being deployed.
   - Persists current, previous and successful image metadata in the private
     `.env`.
   - Starts Postgres and waits for health.
   - Creates and validates a pre-deploy Postgres backup when an existing
     database is running.
   - Runs Drizzle migrations in the migration image.
   - Starts the app and remote backup services.
   - Verifies `/api/health` and `/` through Traefik.
   - Rolls app containers back to the previous image tag automatically when
     migration, restart, application healthcheck or public smoke checks fail.
   - Removes old local `expense-manager-app`, `expense-manager-migrate` and
     `expense-manager-backup` image tags that are no longer needed for the
     current, previous or last successful version.

The public `deploy-summary` artifact contains only service name, image tag,
commit SHA, image owner and a non-reversible hash of the configured host
identifier. It does not substitute for the separately pinned SSH host-key
fingerprint.

Public smoke checks retry for up to three minutes by default because Traefik can
take a short time to route to a recreated container. Override
`PUBLIC_ROUTE_RETRIES` or `PUBLIC_ROUTE_RETRY_DELAY_SECONDS` in the private VPS
`.env` only if your Traefik instance needs a different window.

## VPS Isolation

Use a dedicated `DEPLOY_PATH` for this app, for example
`/mnt/storage/containers/expenses`. Deployment commands run from that directory
with this app's compose file and only manage this app's services, containers
named `expense-manager-*`, and volumes created by this compose project.

The only intentionally shared Docker resource is the external Traefik network
configured by `TRAEFIK_NETWORK` (default `web`). The deploy script does not run
global Docker cleanup commands, so it will not prune images or volumes used by
other applications on the VPS. Image cleanup is scoped to this repository's
three image names and keeps the active tag, previous tag, last successful tag,
last rollback tag and `latest`. Set `IMAGE_CLEANUP_ENABLED=false` only for a
short investigation window when you intentionally want to keep old local app
image tags.

The production compose file mounts required application and database secrets as
Docker Compose secrets instead of passing them directly as service environment
values. The private VPS `.env` still stores deploy-time values with mode `600`,
but `docker inspect` for running app, migration, Postgres and backup containers
does not need to expose `BETTER_AUTH_SECRET`, `POSTGRES_PASSWORD` or
`RESTIC_PASSWORD` as plain environment values.

## Postgres Storage Layout

The production compose files use `postgres:18-alpine`. PostgreSQL 18 stores
cluster data under a major-version-specific `PGDATA` path
(`/var/lib/postgresql/18/docker` in the official image), so the persistent
volume is mounted at `/var/lib/postgresql`.

Do not change the Postgres major version or volume mount path on an existing
database without a tested backup and restore plan.

## Rollback

Rollback has two layers:

- Image rollback: safe to automate. It moves the app and backup containers back
  to a previous image tag and re-runs health and public smoke checks.
- Database restore: intentionally manual. Restoring a pre-deploy dump can remove
  records written after deployment started, so it requires an explicit
  confirmation string.

During deploy, `scripts/deploy-vps.sh` records the previous image tag in
`PREVIOUS_IMAGE_TAG`, creates `LAST_PREDEPLOY_BACKUP` when Postgres already
exists and records `LAST_SUCCESSFUL_IMAGE_TAG` after a successful smoke test. If
the deploy fails after the previous image is known, the script automatically
rolls containers back to `PREVIOUS_IMAGE_TAG`. It does not restore the database
automatically.

For a manual rollback through GitHub:

1. Open `Actions -> Rollback -> Run workflow`.
2. Leave `image_tag` empty to use `PREVIOUS_IMAGE_TAG` from the private VPS
   `.env`, or enter a specific tag such as `sha-abcdef1`.
3. Leave `restore_database_backup` empty for an image-only rollback.
4. To also restore a pre-deploy dump, enter a path such as
   `backups/predeploy/expense_manager-YYYYMMDDTHHMMSSZ-abcdef1.dump` and set
   `confirm_database_restore` to exactly `restore-database`.
5. Approve the protected `production` environment.

Manual rollback from the VPS uses the same script:

```bash
cd /opt/expense-manager

# Image-only rollback to PREVIOUS_IMAGE_TAG from .env.
DEPLOY_PATH=/opt/expense-manager \
COMPOSE_CMD="docker compose" \
bash scripts/rollback-vps.sh

# Image rollback plus explicit database restore from a pre-deploy dump.
IMAGE_TAG=sha-abcdef1 \
RESTORE_DATABASE_BACKUP=backups/predeploy/expense_manager-YYYYMMDDTHHMMSSZ-abcdef1.dump \
CONFIRM_DATABASE_RESTORE=restore-database \
bash scripts/rollback-vps.sh
```

If a migration was destructive or incompatible with the previous app version,
use the database restore option. If the issue is only application startup,
configuration or runtime behavior, prefer image-only rollback first.

For broader disaster scenarios such as a lost VPS, offline NAS, broken
monitoring state or manual database/upload restore, use
[`docs/disaster-recovery.md`](docs/disaster-recovery.md).

## Traefik Expectations

The compose labels assume:

- External Docker network: `web`
- HTTP entrypoint: `web`
- HTTPS entrypoint: `websecure`
- Certificate resolver: `le`

Change `TRAEFIK_NETWORK`, `TRAEFIK_HTTP_ENTRYPOINT`, `TRAEFIK_ENTRYPOINT` and
`TRAEFIK_CERT_RESOLVER` in the VPS `.env` if your Traefik instance uses
different names.

Traefik terminates TLS and forwards traffic to the `app` container on port
`3000`. The application trusts proxy headers only because it is attached to the
private Traefik network and is not published directly to the internet.

## Production Monitoring

The recommended lightweight setup is a separate monitoring compose project on
the VPS, exposed only through Tailscale. Keep it independent from this app's
compose project so other applications on the host are not affected.

At minimum, monitor:

- Public availability: `https://your-domain.example`
- Public health endpoint: `https://your-domain.example/api/health`
- Public DNS resolution and TLS certificate expiry
- Internal health endpoint from the app Docker network:
  `http://app-container-name:3000/api/health`
- Host CPU, memory, disk and filesystem usage through `node_exporter`
- Docker container CPU, memory, filesystem and network usage through cAdvisor
- Authenticated synthetic browser flow using a dedicated test account
- Optional distributed tracing through OTLP HTTP and Grafana Tempo
- Container logs through a private Dozzle instance
- Docker Compose stack management through a private Dockge instance
- Short-retention historical Docker logs through Grafana Loki and Grafana Alloy
- Traefik container presence and restart loops through cAdvisor
- Encrypted application backup freshness and restore-test status
- Encrypted monitoring-stack backup freshness and restore-test status
- Weekly alert delivery fire drill status
- Post-reboot healthcheck status

For the synthetic flow, store the account email and password only as private
files on the VPS or as Docker secrets. The browser check should log in, visit
the critical app routes and expose only Prometheus metrics such as:

```text
expense_manager_synthetic_success
expense_manager_synthetic_duration_seconds
expense_manager_synthetic_last_run_timestamp_seconds
expense_manager_synthetic_route_success
expense_manager_synthetic_route_duration_seconds
```

Useful Prometheus validation queries:

```bash
promtool query instant http://127.0.0.1:9090 \
  'probe_success{app="expenses"}'

promtool query instant http://127.0.0.1:9090 \
  'expense_manager_synthetic_success{app="expenses"}'

promtool query instant http://127.0.0.1:9090 \
  'expense_manager_synthetic_route_success{app="expenses"}'
```

Keep synthetic accounts limited to the minimum role needed for route rendering,
do not reuse personal accounts and never commit monitoring credentials.

For traces, run a Tempo or OpenTelemetry Collector service on a private Docker
network shared with the app. Then set:

```env
OTEL_TRACING_ENABLED=true
OTEL_EXPORTER_OTLP_ENDPOINT=http://tempo:4318
OTEL_TRACES_SAMPLE_RATE=0.1
```

The app records request method, route id, path, status, duration, request id and
trace id. It intentionally does not record request bodies, query strings,
passwords, emails, financial values or customer data.

For container operations, keep management tools private to the tailnet:

- Grafana: `http://<tailscale-ip>:3002`
- Dozzle logs: `http://<tailscale-ip>:3003`
- Dockge compose management: `http://<tailscale-ip>:3004`

Dozzle and Dockge interact with Docker. Treat them as privileged administrative
tools even when they are bound only to Tailscale. Prefer a Docker socket proxy
for read-only consumers such as Dozzle and log collectors. Dockge needs
read-write Docker socket access to manage Compose stacks, so use a strong admin
password and do not expose it through public Traefik routes.

Store generated admin passwords only on the VPS, for example:

```text
/mnt/storage/containers/monitoring/secrets/grafana_admin_password
/mnt/storage/containers/monitoring/secrets/dozzle_admin_password
/mnt/storage/containers/monitoring/secrets/dockge_admin_password
```

Telegram alert delivery should also use VPS-only secret files:

```text
/mnt/storage/containers/monitoring/secrets/telegram_bot_token
/mnt/storage/containers/monitoring/secrets/telegram_chat_id
```

The token is a credential. If it is pasted into a chat, terminal transcript or
ticket, rotate it at the provider and replace the VPS secret file.

Useful container monitoring validation queries:

```bash
promtool query instant http://127.0.0.1:9090 \
  'up{job="cadvisor"}'

promtool query instant http://127.0.0.1:9090 \
  'count(container_last_seen{job="cadvisor",name!=""})'
```

Useful log pipeline validation queries:

```bash
promtool query instant http://127.0.0.1:9090 \
  'up{job=~"loki|alloy"}'

curl -fsS http://loki:3100/loki/api/v1/labels
```

Keep Loki private on the monitoring Docker network. Do not expose Loki, Alloy,
Prometheus or Tempo through public Traefik routes. Use Grafana over Tailscale
to query dashboards and logs.

If a private NAS needs to push Docker logs to the VPS, bind Loki only to the VPS
tailnet IP and keep it out of public Traefik routes. Set a short retention
window such as `LOKI_RETENTION=168h` and configure the NAS log collector to drop
lines older than the same window before forwarding.

For small hosts, keep log retention type-aware: infrastructure/container logs
should stay short, while application logs can have a slightly longer
stream-specific retention if operationally useful. Do not use Loki as permanent
audit storage.

Keep monitoring centralized on the VPS when the VPS is more stable than the NAS.
The NAS should only run lightweight exporters and agents:

- node_exporter for NAS host CPU, memory, disk and filesystem metrics;
- cAdvisor for NAS Docker container metrics;
- Alloy for NAS Docker logs pushed to the VPS Loki endpoint;
- Dozzle for private, on-demand NAS log viewing.

Bind those NAS services only to the NAS tailnet IP and scrape them from the VPS
Prometheus. Do not run a second Grafana/Prometheus stack on the NAS unless the
NAS needs to operate independently during a VPS outage.

Prometheus alert rules should cover public probes, internal health checks,
synthetic browser failures, host memory/disk pressure, cAdvisor scrape health,
Loki/Alloy scrape health, container restart loops, DNS/TLS failures, Traefik,
backup freshness, restore-test freshness, alert fire drills and post-reboot
health. External alert delivery requires a private notification provider
configuration. If using Telegram, keep the bot token and chat id in VPS-only
secret files or protected environment secrets, never in the public repository.

When changing alert rules in this repository, run:

```bash
scripts/ops/test-prometheus-rules.sh
```

After production infrastructure changes, run the read-only operational audit on
the VPS:

```bash
sudo /usr/local/sbin/expense-manager-production-audit
```

For small VPS hosts, keep host security updates automated but conservative. The
recommended baseline is unattended Ubuntu security updates without automatic
reboots. Reboot manually during a maintenance window after checking monitoring,
active sessions and pending kernel updates.

For low-resource hosts, prefer size-capped Docker `json-file` logs plus short
Loki retention over long local log history. A practical baseline is
`max-size=10m`, `max-file=3` for compose-managed containers and `168h` Loki
retention for searchable logs.

## Backup And Restore

Production backups are remote by default. The `backup` service runs daily,
creates temporary files inside the container, uploads them to an encrypted
`restic` repository and then removes local temporary files.

If you do not have a remote backup provider yet, set `BACKUP_ENABLED=false` as a
GitHub production environment variable. This disables the daily remote backup
service and lets the deploy continue without `RESTIC_REPOSITORY` or
`RESTIC_PASSWORD`. Use this only for bootstrap or non-critical data. It is not a
disaster-recovery plan.

The remote repository receives:

- Postgres custom-format dumps.
- Upload archive tarballs.
- SHA-256 checksums.

Use an S3-compatible bucket, Backblaze B2, SFTP, REST server or another restic
backend that is not stored on the same VPS. For S3-compatible storage, set
`RESTIC_REPOSITORY`, `RESTIC_PASSWORD`, `AWS_ACCESS_KEY_ID`,
`AWS_SECRET_ACCESS_KEY` and optionally `AWS_DEFAULT_REGION` in the private VPS
`.env`.

The deploy script also creates a pre-deploy dump under
`backups/predeploy/` before migrations when an existing Postgres container is
running. That local file is only a short-term rollback aid for a deploy; it is
not the disaster-recovery backup.

For a private NAS on the same tailnet, a lightweight alternative is a VPS
`systemd` timer that runs `pg_dump`, validates the dump, archives uploads and
pushes everything to an encrypted restic repository over SFTP:

```text
VPS Postgres/uploads -> restic over SFTP -> NAS /share/Backup/expense-manager/restic
```

This keeps the application compose project independent from the NAS. If the NAS
is offline, the app keeps running, the timer fails, and Prometheus can alert
when the last successful backup becomes stale.

Use `Restart=on-failure` on the backup service for short NAS outages, and run a
separate weekly restore-test timer that restores the latest restic snapshot into
a temporary directory, validates checksums, validates the PostgreSQL dump with
`pg_restore --list` and validates the uploads archive with `tar -tzf`.

Keep the restic password outside the public repository. If it is stored only on
the VPS and the VPS is lost, the NAS repository cannot be decrypted. Store that
password in a private password manager or another secure location.

List remote snapshots:

```bash
docker compose run --rm --no-deps --entrypoint restic backup snapshots
```

Restore a remote snapshot to a temporary restore directory:

```bash
docker compose run --rm --no-deps \
  --entrypoint restic \
  -v "$(pwd)/restore:/restore" \
  backup restore latest --target /restore
```

Restore Postgres first:

```bash
docker compose stop app
docker compose exec -T postgres pg_restore \
  -U "$POSTGRES_USER" \
  -d "$POSTGRES_DB" \
  --clean \
  --if-exists \
  /restore/path/from/restic/expense_manager_YYYYMMDDTHHMMSSZ.dump
```

Restore uploads matching the same timestamp:

```bash
docker compose run --rm --no-deps \
  -v "$(pwd)/restore:/restore:ro" \
  app sh -lc 'rm -rf /app/uploads/* && tar -C /app/uploads -xzf /restore/path/from/restic/uploads_YYYYMMDDTHHMMSSZ.tar.gz'
docker compose up -d app
```

Always validate restores on a separate database before relying on them in
production.

## Public Repository Safety Checklist

- `.env` is ignored and never committed.
- Production values are generic in docs and examples.
- SSH keys are generated on the VPS and stored only in GitHub secrets.
- The deploy workflow does not print the real host value.
- The app is not directly published with Docker ports in the Traefik compose.
- Public route checks use the private VPS `.env` domain at deploy time.
- E2E tests use synthetic users and example domains only.
